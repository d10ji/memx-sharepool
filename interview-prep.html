How to use this doc

The aim isn't to cram trivia — it's to build insider depth. Tools change. Patterns and trade-offs don't.

Drill plan (4 weeks).
Week 1: Foundation + Core (read end-to-end, write the SQL/Python answers by hand).
Week 2: Tools + Architecture (read, then re-explain each "why X over Y" out loud).
Week 3: Project narratives. Read aloud daily, no notes. The 30-second + 3-min versions.
Week 4: Drills only. SQL whiteboard problems timed. Python coding problems timed. Mock interview the project narratives.

Daily loop: 15 minutes — pick one section, read, then close the file and re-explain the section to an imaginary interviewer. If you can't, re-read.

Tone reminders for any answer you give:

  1. Lead with the business problem, not the tools.
  2. Sequence: problem → flow → choice → trade-off → what broke → what I'd do differently.
  3. Never say "we used X" without being ready for "why not Y?"
  4. Be specific with numbers (volume, SLA, latency, cost). Vague = surface. Specific = depth.

SQL fundamentals

SQL absolute basics — the building blocks (start here if rusty)

Every SQL query is built from a few core verbs. If any of these feel shaky, fix them before going deeper.

1. SELECT — pick columns from a table

-- Get everything (avoid in production)
SELECT * FROM customers;

-- Pick specific columns
SELECT customer_id, name, email FROM customers;

-- Rename columns in output
SELECT customer_id AS id, name AS full_name FROM customers;

-- Compute new columns
SELECT name, signup_date, CURRENT_DATE - signup_date AS days_since_signup
FROM customers;

2. WHERE — filter rows

-- Equality, inequality
SELECT * FROM orders WHERE status = 'shipped';
SELECT * FROM orders WHERE amount > 100;
SELECT * FROM orders WHERE amount BETWEEN 10 AND 100;

-- Multiple conditions
SELECT * FROM orders WHERE status = 'shipped' AND amount > 50;
SELECT * FROM orders WHERE region IN ('US', 'CA', 'MX');
SELECT * FROM orders WHERE status <> 'cancelled';

-- Pattern matching
SELECT * FROM customers WHERE email LIKE '%@gmail.com';
SELECT * FROM customers WHERE name LIKE 'A%';   -- starts with A

-- NULL handling (the trap!)
SELECT * FROM customers WHERE phone IS NULL;
-- WRONG: WHERE phone = NULL  (returns no rows, ever)

3. ORDER BY — sort the result

SELECT name, signup_date FROM customers ORDER BY signup_date DESC;
SELECT name, region, signup_date FROM customers ORDER BY region ASC, signup_date DESC;

4. LIMIT / OFFSET — pagination

SELECT * FROM orders ORDER BY order_date DESC LIMIT 10;
SELECT * FROM orders ORDER BY order_date DESC LIMIT 10 OFFSET 20;  -- page 3

5. Aggregates — condense rows into one number

SELECT COUNT(*) FROM orders;                       -- total rows
SELECT COUNT(DISTINCT customer_id) FROM orders;    -- unique customers
SELECT SUM(amount), AVG(amount), MIN(amount), MAX(amount) FROM orders;

6. GROUP BY — aggregate per group

-- Total revenue per region
SELECT region, SUM(amount) AS revenue
FROM orders
GROUP BY region;

-- Filter groups with HAVING (not WHERE!)
SELECT region, SUM(amount) AS revenue
FROM orders
GROUP BY region
HAVING SUM(amount) > 1000;

7. JOIN — combine two tables

-- Each order with customer name
SELECT o.order_id, o.amount, c.name
FROM orders o
JOIN customers c ON o.customer_id = c.customer_id;

8. Combining everything — a real query

-- Top 5 regions by total revenue in 2025, only paid orders
SELECT
    c.region,
    COUNT(DISTINCT o.customer_id) AS customers,
    SUM(o.amount)                  AS revenue
FROM orders o
JOIN customers c USING (customer_id)
WHERE o.status = 'paid'
  AND o.order_date >= DATE '2025-01-01'
  AND o.order_date <  DATE '2026-01-01'
GROUP BY c.region
HAVING SUM(o.amount) > 10000
ORDER BY revenue DESC
LIMIT 5;

If you can read and write that query without effort, you've got the basics. Now go deeper.

JOIN semantics — what each one actually does
  • INNER JOIN — rows where the key matches in both tables.
  • LEFT JOIN — all rows from left, matching from right or NULL. This is your default for fact ↔ dim.
  • RIGHT JOIN — rarely used; flip the order and use LEFT instead for readability.
  • FULL OUTER JOIN — everything from both, NULL where missing. Useful for reconciliation.
  • CROSS JOIN — cartesian product. Useful for date spines, calendar generation.
  • SEMI JOIN / ANTI JOIN — "rows in A that have/don't have a match in B". Snowflake/Spark have explicit syntax; in standard SQL use EXISTS / NOT EXISTS.
Trap: Filtering in WHERE vs ON in a LEFT JOIN gives different results. WHERE b.x = 5 turns it into INNER JOIN; put the filter in ON if you want LEFT semantics preserved.
Real example from healthcare claims work

In the claims pipeline, every claim line LEFT JOINs dim_provider three times — once each for rendering, billing, and facility provider. We have to use LEFT (not INNER) because some claims don't have all three NPIs. Putting WHERE rendering_provider.in_network = TRUE would silently drop claims with missing rendering NPIs — a real bug we caught in code review. Network status filters live in ON, not WHERE.

GROUP BY semantics — what gets returned

One row per unique combination of GROUP BY columns. Every column in SELECT must be either in GROUP BY or inside an aggregate.

SELECT customer_id, COUNT(*) AS orders, SUM(amount) AS revenue
FROM fct_orders
WHERE order_date >= '2025-01-01'
GROUP BY customer_id
HAVING COUNT(*) > 5;

Order of execution (memorize): FROM → JOIN → WHERE → GROUP BY → HAVING → SELECT → DISTINCT → ORDER BY → LIMIT.

Why it matters: aliases defined in SELECT are not available in WHERE (because SELECT runs after). Some engines (BigQuery, DuckDB) allow it; standard SQL doesn't.

Window functions — the workhorse

Aggregate without collapsing rows. The shape:

FUNC(...) OVER (PARTITION BY ... ORDER BY ... ROWS/RANGE BETWEEN ... AND ...)

The five you need cold:

  • ROW_NUMBER() — deduping, top-N per group.
  • RANK() / DENSE_RANK() — ranking with tie behavior.
  • LAG() / LEAD() — previous/next row. For deltas, sessionization.
  • SUM() OVER / AVG() OVER — running totals, moving averages.
  • FIRST_VALUE / LAST_VALUE — pick first/last in window.
-- Latest row per customer (dedup)
SELECT *
FROM (
  SELECT *, ROW_NUMBER() OVER (PARTITION BY customer_id ORDER BY updated_at DESC) AS rn
  FROM raw.customer
)
WHERE rn = 1;
Real example from healthcare claims work

This pattern is the heart of claim line dedup. PARTITION BY is much wider for claims because the "same row" is defined by a clinical fingerprint:

ROW_NUMBER() OVER (
  PARTITION BY claim_id, claim_line_number, person_id,
               claim_line_start_date, claim_line_end_date,
               COALESCE(place_of_service_code, ''),
               COALESCE(revenue_center_code, ''),
               COALESCE(hcpcs_code, ''),
               COALESCE(rendering_npi, ''),
               COALESCE(paid_amount, 0.0),
               COALESCE(allowed_amount, 0.0),
               COALESCE(charge_amount, 0.0)
  ORDER BY paid_date DESC, paid_amount DESC, allowed_amount DESC
) AS rn
WHERE rn = 1;

The COALESCE on every column is critical — NULL = NULL is NULL, so without COALESCE two rows with NULL revenue codes would land in different partitions. The ORDER BY uses paid_date first because for re-adjudicated claims, the latest payment is the truth.

CTEs — readable, sometimes optimization-fenced 
WITH recent_orders AS (
  SELECT * FROM fct_orders WHERE order_date >= CURRENT_DATE - 30
),
cust_totals AS (
  SELECT customer_id, SUM(amount) AS total_30d FROM recent_orders GROUP BY 1
)
SELECT c.name, t.total_30d
FROM dim_customer c
JOIN cust_totals t USING (customer_id);

Note: Postgres treats CTEs as optimization fences (materialized) by default before v12; Snowflake/BigQuery inline them. Don't assume the planner inlines.

Recursive CTE — for hierarchies (org charts, category trees). Memorize the anchor + recursive-step + UNION ALL pattern.

Real example from healthcare claims work

Real dbt staging models in the claims pipeline use chained CTEs for readability. The dedup model is roughly:

WITH null_claim_type_fixing AS (
  -- attach normalized claim type (UB / HCFA), build new_claim_id
  SELECT ...
),
claims_with_multiple_dates AS (
  -- detect claims spanning multiple service dates
  SELECT split_part(new_claim_id,'_',1) AS claim_id, ...
  FROM null_claim_type_fixing
  GROUP BY 1
  HAVING COUNT(DISTINCT claim_start_date) > 1
),
new_claim_id_for_multiple_dates AS (
  -- generate composite claim_id for these
  SELECT ...
),
complete_duplicate_claims AS (
  -- ROW_NUMBER for dedup on clinical fingerprint
  SELECT *, ROW_NUMBER() OVER (...) AS rn
  FROM new_claim_id_for_multiple_dates
)
SELECT * FROM complete_duplicate_claims;

Each CTE has one job: name fixing → date detection → ID composition → dedup. Anyone reading the model can trace it top-to-bottom. Without CTEs this would be a 200-line subquery soup.

Date functions — you'll use these every day

Syntax varies by warehouse. The concepts don't.

-- TRUNCATE to a unit (start of day/month/quarter/year)
DATE_TRUNC('day', event_ts)        -- 2025-04-28 00:00:00
DATE_TRUNC('month', event_ts)      -- 2025-04-01 00:00:00
DATE_TRUNC('quarter', event_ts)    -- 2025-04-01 00:00:00
DATE_TRUNC('year', event_ts)       -- 2025-01-01 00:00:00

-- ADD an interval
DATEADD(day, 7, order_date)        -- 7 days later (Snowflake)
order_date + INTERVAL '7 days'     -- Postgres syntax
DATE_ADD(order_date, INTERVAL 7 DAY) -- BigQuery / MySQL

-- DIFFERENCE between two timestamps
DATEDIFF(day, signup_date, first_order_date)        -- Snowflake
EXTRACT(epoch FROM (b - a)) / 86400                 -- Postgres (days)
TIMESTAMPDIFF(HOUR, a, b)                           -- MySQL

-- EXTRACT a part
EXTRACT(year FROM event_ts)        -- 2025
EXTRACT(dow FROM event_ts)         -- day of week (0=Sun)
EXTRACT(hour FROM event_ts)        -- 14

-- CURRENT date / time
CURRENT_DATE, CURRENT_TIMESTAMP, CURRENT_TIME

-- FORMAT to string
TO_CHAR(event_ts, 'YYYY-MM-DD')

Common patterns:

-- Orders per day
SELECT DATE_TRUNC('day', order_ts) AS day, COUNT(*)
FROM orders GROUP BY 1 ORDER BY 1;

-- Last 30 days
WHERE order_date >= CURRENT_DATE - INTERVAL '30 days'

-- This month vs last month
WHERE DATE_TRUNC('month', order_date) IN (
  DATE_TRUNC('month', CURRENT_DATE),
  DATE_TRUNC('month', CURRENT_DATE - INTERVAL '1 month')
)

-- Day-of-week pattern (Mondays only)
WHERE EXTRACT(dow FROM order_ts) = 1
Real example from healthcare claims work

Claims have 5+ relevant dates: claim_start_date, claim_end_date, claim_line_start_date, paid_date, received_date, plus admission_date and discharge_date for inpatient stays. Two real patterns:

-- 1. Reconstruct claim spans for inpatient claims (multiple line dates)
SELECT claim_id,
       MIN(claim_start_date) AS min_start_date,
       MAX(claim_end_date)   AS max_end_date
FROM raw.medical_claim
GROUP BY claim_id
HAVING COUNT(DISTINCT claim_start_date) > 1;

-- 2. Length of stay (LOS) for inpatient
SELECT claim_id,
       DATEDIFF(day, admission_date, discharge_date) AS los_days
FROM medical_claim
WHERE claim_type = 'institutional' AND admission_date IS NOT NULL;

Real bug we hit: 2025-06-20 00:00:00 from upstream parsed as 1900-01-01 because the mapping tool didn't recognize the timestamp format. Always validate paid_date isn't 1900-01-01 before trusting it.

String functions — cleaning data is half the job 
-- Concat
'Hello ' || name                -- Postgres / Snowflake / Oracle
CONCAT('Hello ', name)          -- portable

-- Length / case
LENGTH(name), CHAR_LENGTH(name)
LOWER(email), UPPER(country_code)

-- Substring
SUBSTRING(name, 1, 3)           -- first 3 chars
SUBSTR(name, -4)                -- last 4 (some warehouses)
LEFT(name, 3), RIGHT(name, 4)

-- Replace, trim
REPLACE(phone, '-', '')         -- strip dashes
TRIM(name)                       -- strip leading/trailing whitespace
LTRIM(s), RTRIM(s)
REGEXP_REPLACE(phone, '[^0-9]', '', 'g')  -- digits only

-- Split + index
SPLIT_PART(email, '@', 2)       -- domain part
SPLIT(s, ',')                    -- returns array

-- Contains / starts / ends
name LIKE 'A%'                   -- starts with A
name ILIKE '%alice%'             -- case-insensitive contains (Postgres)
POSITION('@' IN email)           -- char position
REGEXP_LIKE(email, '^[a-z]+@[a-z]+\\.[a-z]+$')

Real-world cleaning:

-- Normalize emails for matching
SELECT LOWER(TRIM(email)) AS email_clean
FROM customers;

-- Extract domain
SELECT SPLIT_PART(LOWER(email), '@', 2) AS domain, COUNT(*)
FROM customers GROUP BY 1 ORDER BY 2 DESC;

-- Mask PII
SELECT
  LEFT(email, 2) || '***' || SPLIT_PART(email, '@', 2) AS email_masked

Real example from healthcare claims work

Provider Tax ID (TIN) cleaning is a string-function workout. Source data has dashes, leading-zero issues, and 7- or 8-digit values that need padding to 9. Real macro from production:

-- Clean billing TIN: trim, remove dashes, pad to 9 digits if 7-8 chars
CASE
  WHEN LEN(TRIM(REPLACE(billing_tin, '-', ''))) IN (7, 8)
    THEN RIGHT('00' || TRIM(REPLACE(billing_tin, '-', '')), 9)
  ELSE TRIM(REPLACE(billing_tin, '-', ''))
END AS billing_tin

-- Clean NPI: trim, nullif empty
NULLIF(TRIM(rendering_npi), '') AS rendering_npi

-- Provider name: lowercase, trim, nullif empty (called via custom dbt macro)
{{ clean_provider_field('c.rendering_provider_first_name', 'rendering_provider_first_name') }}

Why pad to 9 digits? Some payers strip leading zeros from TINs (treating them as integers somewhere upstream). A real TIN 007654321 arrives as 7654321. Without the pad, you can't join to provider master data.

CASE WHEN — the conditional you'll write 100 times 
-- Bucket numeric values
SELECT
  CASE
    WHEN amount < 10  THEN 'small'
    WHEN amount < 100 THEN 'medium'
    WHEN amount < 1000 THEN 'large'
    ELSE 'enterprise'
  END AS order_size,
  COUNT(*) AS n
FROM orders
GROUP BY 1;

-- Conditional aggregation (a workhorse)
SELECT
  customer_id,
  COUNT(*) AS total_orders,
  SUM(CASE WHEN status = 'paid' THEN 1 ELSE 0 END)         AS paid_orders,
  SUM(CASE WHEN status = 'paid' THEN amount ELSE 0 END)    AS paid_revenue,
  AVG(CASE WHEN status = 'paid' THEN amount END)           AS avg_paid_amt
FROM orders
GROUP BY customer_id;

-- Pivot: months as columns
SELECT
  customer_id,
  SUM(CASE WHEN MONTH(order_date)=1 THEN amount END) AS jan,
  SUM(CASE WHEN MONTH(order_date)=2 THEN amount END) AS feb
FROM orders
GROUP BY customer_id;
Pro tip: SUM(CASE WHEN x THEN 1 ELSE 0 END) is also written as COUNT(CASE WHEN x THEN 1 END) — either works. The COUNT form is slightly cleaner.

  WHEN paid_amount < 100     THEN 'low'

  WHEN paid_amount < 10000   THEN 'high'
  ELSE 'catastrophic'
END AS spend_bucket
NULL handling — the silent killer
  • NULL = NULL is NULL, not TRUE. Use IS NULL / IS NOT NULL.
  • NOT IN (NULL, ...) always returns NULL. Use NOT EXISTS instead.
  • COUNT(col) ignores NULLs. COUNT(*) doesn't.
  • SUM(NULL) over an all-null group returns NULL, not 0. Use COALESCE(SUM(col), 0) if you want 0.
Real example from healthcare claims work

Healthcare data is full of NULLs — missing NPIs, optional procedure codes, partial provider names. The staging-layer pattern:

-- Fall through identifiers
COALESCE(NULLIF(TRIM(person_id), ''),
         NULLIF(TRIM(member_id), ''),
         'Unknown') AS person_id

-- Default text columns to literal sentinel (avoids breaking joins)
COALESCE(payer, 'Unknown') AS payer
COALESCE(plan,  'Unknown') AS plan

-- COALESCE inside PARTITION BY for dedup — NULL would split partitions otherwise
PARTITION BY claim_id, COALESCE(revenue_center_code, ''),
             COALESCE(hcpcs_code, ''), COALESCE(rendering_npi, '')

The 'Unknown' sentinel pattern is a real interview talking point: it preserves NULL semantics for downstream filtering (you can WHERE person_id != 'Unknown') without breaking joins or aggregates that NULL would.

UNION, UNION ALL, INTERSECT, EXCEPT

Set operations stack rows from multiple SELECTs. Column count and types must match.

-- UNION removes duplicates (slower)
SELECT email FROM customers
UNION
SELECT email FROM leads;

-- UNION ALL keeps duplicates (faster, default choice unless you need dedup)
SELECT 'order'  AS event_type, customer_id, event_ts FROM orders
UNION ALL
SELECT 'login'  AS event_type, customer_id, event_ts FROM logins
UNION ALL
SELECT 'logout' AS event_type, customer_id, event_ts FROM logouts;

-- Difference / intersection
SELECT customer_id FROM orders
EXCEPT
SELECT customer_id FROM refunds;     -- customers who never refunded

SELECT customer_id FROM premium
INTERSECT
SELECT customer_id FROM active;      -- premium AND active
Default to UNION ALL. UNION does an extra dedup pass. Use UNION only if you actually need dedup.
Real example from healthcare claims work

EXCEPT is the data-quality reconciliation tool of choice. The pipeline has an observability model that compares source vs final:

-- Claims excluded from final output (with reasons)
SELECT 'eligibility_excluded' AS reason, claim_id FROM raw.medical_claim
EXCEPT
SELECT 'kept' AS reason, claim_id FROM final.medical_claim;

-- Member counts: source vs final (control totals)
SELECT 'source_only' AS where_from, COUNT(DISTINCT member_id) AS n FROM (
  SELECT member_id FROM raw.medical_claim
  EXCEPT SELECT member_id FROM final.medical_claim
) t
UNION ALL
SELECT 'final_only', COUNT(DISTINCT member_id) FROM (
  SELECT member_id FROM final.medical_claim
  EXCEPT SELECT member_id FROM raw.medical_claim
) t;

Stakeholders ask "where did claim X go?" all the time. EXCEPT-based observability gives the answer in one query.

EXISTS / NOT EXISTS / IN / NOT IN — subqueries 
-- Customers who have at least one order (correlated subquery)
SELECT *
FROM customers c
WHERE EXISTS (
  SELECT 1 FROM orders o WHERE o.customer_id = c.customer_id
);

-- Same thing with IN (works, but EXISTS is preferred for big subquery)
SELECT * FROM customers
WHERE customer_id IN (SELECT customer_id FROM orders);

-- Customers who never ordered — PREFER NOT EXISTS
SELECT *
FROM customers c
WHERE NOT EXISTS (
  SELECT 1 FROM orders o WHERE o.customer_id = c.customer_id
);

-- DON'T use NOT IN with possible NULLs — returns no rows!
-- WHERE customer_id NOT IN (SELECT customer_id FROM blocked)
-- If `blocked` has any NULL, the entire result is empty.
Indexes (OLTP) vs micro-partitions / clustering (OLAP)

OLTP DBs (Postgres, MySQL): B-tree indexes for point lookups, GIN for full-text/array.

Cloud warehouses don't use traditional indexes. They use:

  • Snowflake: micro-partitions (auto), clustering keys (manual hint for queries with predictable filter patterns).
  • Redshift: dist-key + sort-key (you choose).
  • BigQuery: partitioning + clustering.

The shared idea: minimize bytes scanned. Partition pruning is the single biggest perf lever in OLAP — queries that don't filter the partition column scan everything.

6. Putting it together — a real DE warm-up

"""Read orders.csv, group by customer, output top 5 by revenue."""
import csv
from collections import defaultdict

totals = defaultdict(float)
with open("orders.csv") as f:
    for row in csv.DictReader(f):
        totals[row["customer_id"]] += float(row["amount"])

top5 = sorted(totals.items(), key=lambda kv: kv[1], reverse=True)[:5]
for cust_id, rev in top5:
    print(f"{cust_id}: ${rev:,.2f}")

If you can write that without looking it up, you're ready. If not, drill until you can.

Real example from healthcare claims work

Equivalent claims-flavored warm-up — reading raw claims and bucketing by procedure:

"""Read claims.csv, count claim lines per HCPCS code, top 10."""
import csv
from collections import Counter

counter = Counter()
with open("claims.csv") as f:
    for row in csv.DictReader(f):
        if row.get("hcpcs_code"):                  # skip nulls

The 1900-01-01 default is a real production bug: when an upstream tool can't parse 2025-06-20 00:00:00 (timestamp format), it silently writes 1900-01-01 as the paid_date. This propagates downstream and causes claims to look "ancient" in finalization. Defensive parsing:

from datetime import datetime, date

EPOCH_GUARD = date(1901, 1, 1)   # anything before this is suspect

def parse_paid_date(s: str) -> date | None:
    if not s or s.strip() == "":
        return None
    for fmt in ("%Y-%m-%d %H:%M:%S", "%Y-%m-%d", "%m/%d/%Y"):
        try:
            d = datetime.strptime(s.strip(), fmt).date()
            if d < EPOCH_GUARD:
                return None    # treat 1900-01-01 as missing, not a real date
            return d
        except ValueError:
            continue
    return None

Data modeling basics

Star vs Snowflake schema

Star: one fact table, dimensions hang off directly. Denormalized dims. Fast queries (one join). The default for analytics.

Snowflake: dimensions are normalized (split across multiple tables). More joins, less redundancy. Use only when dimension storage is huge or write-heavy.

For 95% of analytics warehouses: star. The "snowflake" schema is named after the shape, unrelated to the warehouse vendor.

Fact tables — grain matters more than anything

Define the grain first: "one row per X". E.g. "one row per order line item per day". Grain dictates everything — what dimensions apply, what aggregates make sense, what duplicates look like.

Types:

  • Transaction fact: one row per event (order, click). Most common.
  • Periodic snapshot: one row per entity per period (account balance per day).
  • Accumulating snapshot: one row per entity, columns updated as it progresses through stages (order → shipped → delivered).

Anti-joins, self-joins, lateral joins (the patterns that confuse juniors)

Anti-join — "rows in A with no match in B"

-- Customers who never placed an order
SELECT c.*
FROM customers c
LEFT JOIN orders o ON c.customer_id = o.customer_id
WHERE o.customer_id IS NULL;

-- Same with NOT EXISTS (often clearer / faster)
SELECT c.*
FROM customers c
WHERE NOT EXISTS (
  SELECT 1 FROM orders o WHERE o.customer_id = c.customer_id
);

Self-join — same table, two roles

-- Employees and their managers
SELECT
  e.id   AS emp_id,
  e.name AS emp_name,
  m.name AS manager_name
FROM employees e
LEFT JOIN employees m ON e.manager_id = m.id;

-- Find pairs of customers with same email domain
SELECT a.customer_id, b.customer_id
FROM customers a
JOIN customers b ON SPLIT_PART(a.email, '@', 2) = SPLIT_PART(b.email, '@', 2)
                 AND a.customer_id < b.customer_id;  -- no duplicates / self-pairs

LATERAL join — "for each row in A, run a subquery"

-- For each customer, get their 3 most recent orders
SELECT c.customer_id, c.name, o.order_id, o.amount, o.order_ts
FROM customers c,
LATERAL (
  SELECT order_id, amount, order_ts
  FROM orders
  WHERE customer_id = c.customer_id
  ORDER BY order_ts DESC
  LIMIT 3
) o;

LATERAL is in Postgres / Snowflake / Redshift; in BigQuery use UNNEST or correlated subqueries.

Self-join builds adjustment chains for claim reversals. Each claim has an optional adjusts_claim_id pointing to the claim it modifies. Walking the chain:

-- Direct self-join: each claim with the claim it adjusts
SELECT
  c.claim_id      AS this_claim,
  c.paid_amount   AS this_paid,
  o.claim_id      AS adjusts_original,
  o.paid_amount   AS original_paid
FROM medical_claim c
LEFT JOIN medical_claim o ON c.adjusts_claim_id = o.claim_id;

-- Recursive: walk the entire adjustment chain
WITH RECURSIVE chain AS (
  SELECT claim_id, adjusts_claim_id, 0 AS depth
  FROM medical_claim
  WHERE adjusts_claim_id IS NULL  -- root claims
  UNION ALL
  SELECT c.claim_id, c.adjusts_claim_id, ch.depth + 1
  FROM medical_claim c
  JOIN chain ch ON c.adjusts_claim_id = ch.claim_id
)
SELECT * FROM chain WHERE depth > 0;   -- only adjustments

Query plans — what to look for

Run EXPLAIN (or EXPLAIN ANALYZE) before optimizing. Things that should make you flinch:

  • Full table scan on a large table when a filter exists — missing index/partition.
  • Hash join when one side is small — could be a broadcast join (Spark) or nested loop.
  • High row estimates mismatching actual — stale stats, run ANALYZE.
  • Spills to disk — sort/group by exceeds memory; rewrite or scale up.
  • Cartesian product — missing or wrong join condition.

Snowflake: query profile shows partitions scanned vs total. scanned/total > 50% on a filtered query means the cluster key isn't helping — check predicate alignment.

When to denormalize (and when not to)

Denormalize when: read-heavy, predictable join patterns, BI tools that struggle with joins, query latency budget tight.

Don't denormalize when: dimension data updates frequently (every read joins to current state anyway), data changes need to propagate to many tables (write amplification).

Modern warehouses: denormalize aggressively at the mart layer. Wide tables (`customer_360` with 60 columns) are normal. Storage is cheap, joins at query time are expensive.

Why some queries are slow — the standard checklist
  1. Did the query touch all partitions? Filter on the partition column.
  2. Is there a clustering key? Are the predicates aligned with it?
  3. Big join with no filter? Push filters before the join.
  4. Wildcard SELECT *? List columns; column pruning helps massively in Parquet/Snowflake.
  5. Stale stats? Postgres/Redshift: ANALYZE. Snowflake: auto.
  6. Is the result actually large? Maybe the query is fine but the cardinality is huge — pre-aggregate.
  7. UDFs / non-SARGable predicates? WHERE LOWER(email) = 'x' kills index usage; use functional index or normalize on write.
Real example from healthcare claims work

The provider voting model was the slowest in the pipeline — 30+ minutes on large tenants. The slow query was self-joining the claims table to itself for NPI scoring. Diagnosis steps:

  1. Snowflake query profile showed 80% of time in a single shuffle — the self-join across all claim lines.
  2. The WHERE rendering_npi IS NOT NULL filter was being applied AFTER the join (planner's choice).
  3. Pushed the filter into a CTE that ran first; rewrote the self-join as a window function (COUNT(*) OVER (PARTITION BY claim_id, rendering_npi)).
  4. Cut runtime to ~5 minutes.

Lesson: when SQL is slow, look for shuffles in the query profile first. Then check whether you can replace a self-join with a window function — same result, often dramatically faster because no shuffle.

The honest hierarchy: SQL on the warehouse > DuckDB > Polars > pandas > Spark. Reach for Spark only when you must.

Pipeline patterns (the part that makes you a DE, not a Python dev)

Idempotency — the most important word

An idempotent pipeline produces the same result whether it runs once or ten times. This is what lets you retry safely.

Patterns:

  • Overwrite-by-partition: each run drops + rewrites a date partition. Re-run = same result.
  • Insert-with-uniqueness: dedupe by an idempotency key on insert (MERGE on a natural key).
  • Output a new version: write to v=20250428T0900 and atomic rename. Old versions stay until cleanup.
Anti-pattern: "INSERT INTO" without a uniqueness check. Retries = duplicates. Always.
Real example from healthcare claims work

The claims pipeline is idempotent at three layers:

  1. SQS retry semantics. If the orchestrator fails partway through, SQS re-delivers the message. Same prospect_id + same source data → same output, every time.
  2. dbt build is naturally idempotent. Each model is a SELECT; running again rebuilds the table from source. dbt build for prospect X today produces the same final tables as dbt build tomorrow if the source hasn't changed.
  3. Per-tenant DB recreation. The orchestrator does CREATE DATABASE IF NOT EXISTS {ENV}_INGEST_{prospect_id} — safe to call repeatedly.

This is why we can re-queue a failed ingest with retry_count + 1 without worry — even if the previous attempt got halfway through dbt build, the next attempt rebuilds cleanly.

Watermarks and late-arriving data

A watermark is "I've processed all events with timestamp ≤ T". Anything before T is considered settled.

Patterns:

  • Tumbling window with grace period: process daily, but include events up to 24h late.
  • Append-only with reprocessing window: dbt incremental with WHERE event_date >= (max in target) - 2 days.
  • Mark-and-correct: late records flagged, downstream marts recompute the affected window.

The trade-off: tighter watermark = lower latency but more reprocessing churn. Looser watermark = higher latency but more stable downstream.

Real example from healthcare claims work

Re-adjudicated claims are the canonical late-arriving data problem. A claim originally paid in March can arrive again in November with a different paid amount — that's an 8-month "lateness". Two patterns we used:

  • Per-tenant full rebuild: each ingest event runs the entire pipeline for that tenant. The "watermark" is essentially the latest ingest_datetime — everything before that is settled, then we replace it. Idempotent and simple.
  • Cross-tenant export incremental: the macro-export model that aggregates across tenants uses WHERE ingest_datetime >= (latest in target) - INTERVAL '7 days'. The 7-day buffer catches retroactive corrections we might have missed.

If we'd built incremental on the per-tenant pipeline (instead of full rebuild), we'd have had to handle re-adjudications by detecting the change and re-running the affected window — significantly more complex and bug-prone.

Full vs incremental load

Full: every run reads all source data, rebuilds target. Simple, slow, expensive at scale.

Incremental: every run reads only new/changed source data, MERGEs into target. Complex, fast, error-prone if dedup logic wrong.

Heuristic:

  • Full refresh if the table is < 1M rows or runs in < 5 min.
  • Incremental if it's larger or longer.
  • Always keep a full-refresh path for backfill.

dbt incremental strategies:

  • append — pure insert, dedup downstream.
  • merge — MERGE on unique key. Default for most warehouses.
  • delete+insert — delete affected partition, insert new. Snowflake, Redshift.
  • insert_overwrite — partition-level replace. BigQuery, Spark.
Exactly-once vs at-least-once vs at-most-once
  • At-most-once: events may be lost, never duplicated. Rare.
  • At-least-once: events may be duplicated. The common case. Combine with idempotent processing for de facto exactly-once.
  • Exactly-once: each event processed exactly once. Hard. Often "exactly-once semantics" via idempotency + transactional commits, not literally exactly-once.
Practical rule: design for at-least-once delivery, idempotent processing. That gives you exactly-once outcomes without the complexity.
Backfilling — how to recompute history without breaking prod

Backfill = re-run the pipeline for a past date range. You'll do this for: bug fixes, schema changes, new dimensions, late data corrections.

The non-negotiables:

  • Idempotency. Re-running for any date must produce the same result. (Already covered. Without this, no backfill.)
  • Date-aware code. Tasks accept a logical date parameter. {{ ds }} in Airflow, --ds in your CLI.
  • Bounded windows. Backfill one day at a time, in parallel where safe. Don't run a 6-month backfill in one query.
  • Quarantine path. If backfill is risky, run into a separate schema first, validate, then swap.
# Airflow: catchup or manual backfill

PySpark

When Spark is overkill (and what to use instead)

Spark is the right answer when:

  • Your data doesn't fit on one machine (> ~100GB working set).
  • You need distributed shuffles (joins of huge tables).
  • You're already on a Spark stack (Databricks, EMR, Glue) and adding workload.

It's overkill when:

  • Data fits in Snowflake / BigQuery / DuckDB. SQL on the warehouse beats Spark on cost and ops.
  • Data fits in Polars on one machine. Polars is faster than Spark for < ~100GB.
  • Cluster spin-up time exceeds the actual compute. (Glue cold-start is ~1 min — meaningful for short jobs.)
Partitioning, broadcast joins, skew, AQE

Partitioning: df.repartition(n, "key") for shuffle, df.write.partitionBy("date") for storage layout. Default partition count: spark.sql.shuffle.partitions = 200 — tune for cluster size and data volume.

Broadcast join: when one side is small (< 100MB), broadcast it. Spark does this automatically up to spark.sql.autoBroadcastJoinThreshold; force with broadcast(small_df).

Skew: one key dominates a join, one task takes 90% of the time. Symptoms: long tail in the Stages UI. Fixes:

  • Salting: append a random suffix to the skewed key, expand on the other side.
  • AQE (Adaptive Query Execution, Spark 3+): handles skew automatically if enabled. Turn it on: spark.sql.adaptive.enabled = true.

Incremental strategies — pick with intent
  • merge (default in Snowflake/BigQuery): MERGE on unique_key. Updates existing, inserts new. Best for SCD or when records can update.
  • delete+insert (Snowflake, Redshift): DELETE matching keys, INSERT new. Faster than MERGE for large incremental windows.
  • insert_overwrite (BigQuery, Spark): replace whole partitions. Best when partitions are atomic units.
  • append: pure INSERT, dedup downstream. Use only when source is append-only and you have separate dedup logic.
Real example from healthcare claims work

The claims pipeline mostly uses full rebuild per tenant per ingest, because:

  • Each ingest event is a new dataset version — not a delta on the previous.
  • Per-tenant volume is bounded (typical: 1-50M claim lines per tenant).
  • Re-adjudicated claims would make incremental dedup complicated.
  • Snowflake is fast enough at this scale that incremental complexity isn't worth the savings.

Where we did go incremental: cross-tenant aggregation extracts spanning many ingest events. There we use merge with unique_key='claim_row_id' and WHERE ingest_datetime >= ... to scope the read.

Interview takeaway: full vs incremental isn't always the obvious "incremental for big tables" answer — sometimes the data shape (re-adjudication) makes full rebuild simpler and safer.

  • Version control: SQL in git, code review, branching.

Airflow

DAG design principles
  • One DAG per business domain, not one DAG per source table. Enough granularity for ownership, not so much that scheduling becomes chaotic.
  • Tasks are idempotent. Retrying a task should produce the same result.
  • Tasks are atomic. Either fully succeed or fully fail. No half-done state.
  • Use templating for dates: {{ ds }} is the run date. Pass it through to the underlying SQL/script. Lets you backfill any date.
  • Avoid heavy work on the scheduler: don't query a DB or call an API in your DAG file. It runs every parse cycle (~30s).
Sensors vs deferrable operators

Old-school sensors: S3KeySensor, ExternalTaskSensor. They occupy a worker slot while polling. Expensive at scale.

Deferrable operators (Airflow 2.2+): yield to a triggerer process while waiting. Free up worker slots. Use these for any sensor that waits longer than a minute.

from airflow.providers.amazon.aws.sensors.s3 import S3KeySensorAsync

wait = S3KeySensorAsync(
    task_id="wait_for_file",
    bucket_key="s3://bucket/data/{{ ds }}/_SUCCESS",
)
XCom anti-patterns
  • Don't push large data through XCom. XCom is metadata storage in the metadata DB. Pushing a DataFrame is a recipe for an explosion. Push pointers (S3 paths) instead.
  • Don't use XCom for control flow. Use BranchPythonOperator or TaskGroup.
  • Don't share state via XCom across DAGs. Use a database, S3, or a proper message bus.

Lambda — data engineering use cases
  • Lightweight ingestion: API webhook receiver writing to Kinesis.
  • S3 event triggers: file lands → Lambda kicks off downstream task.
  • Validation/enrichment on streaming data.

Step Functions vs Airflow

Step Functions: state machine, JSON-defined, native AWS service integrations, event-driven. Best for: stateless workflows, Lambda-heavy fan-outs, simple linear pipelines.

Airflow: Python-defined DAGs, sensors, complex inter-task dependencies, mature plugin ecosystem. Best for: data pipelines with complex dependencies, team familiar with Python.

The split in practice: Step Functions for the "kick off and orchestrate AWS services" layer; Airflow for the "data pipeline" layer. Often coexist.

Athena, DMS, Kinesis, IAM — the rest of the toolbox
  • Athena: serverless SQL on S3 via Glue Catalog. Ad-hoc queries, light analytics. Pay per TB scanned. Use partitioning and Parquet to control cost.
  • DMS: Database Migration Service. Full + ongoing CDC from RDBMS to S3 or another DB. Managed alternative to Debezium.
  • Kinesis: Data Streams (Kafka analog), Firehose (delivery to S3/Redshift), Data Analytics (Flink-managed). Pick Streams for buffering, Firehose for delivery.
  • IAM for DEs: minimum-privilege roles per service, no long-lived keys (use roles for EC2/Lambda/Glue), service-linked roles where possible. Don't share an admin role across all your jobs.

Snowflake

Virtual warehouses — the right mental model

A warehouse is a compute cluster. Storage is separate. Multiple warehouses can read the same data without contention.

  • Sizes: XS, S, M, L, XL, ... (each step doubles cost and ~doubles speed).
  • Auto-suspend: warehouse spins down after N seconds idle. Set to 60s — per-second billing means it's almost free to resume.
  • Multi-cluster: warehouse can scale out to multiple clusters under load (auto-scaling). Use for variable concurrent BI workloads.
  • Warehouse per workload: dbt has its own, BI has its own, ETL has its own. Isolation + cost attribution.
Real example from healthcare claims work

The claims pipeline uses per-tenant database isolation rather than a single shared warehouse with row-level security. The naming convention encodes everything:

{ENV}_INGEST_{PROSPECT_ID}    -- one DB per tenant
   e.g. MARE_DEV_INGEST_112_DATASET
        MARE_DEMO_INGEST_AXA_DATASET
        MARE_OFFSHORE_INGEST__DATASET

{ENV}_INPUT_LAYER             -- shared raw landing (multiple input_layer_{prospect_id} schemas)
{ENV}_TUVA_DEMO_SEED          -- shared reference data (DRG, ICD, CPT)

Why DB-level not schema-level isolation? Compliance. A "DROP DATABASE" gives clean removal of a tenant; permission-misconfiguration risk is much lower with separate DBs than with shared DBs and RLS.

Micro-partitions, clustering, pruning

Snowflake auto-partitions data into ~16MB compressed micro-partitions. Each has min/max stats per column.

Pruning: query filters use stats to skip partitions. This is the biggest perf lever.

Clustering keys: explicit hint to keep similar values in the same micro-partitions. Set when:

  • Table is large (> 1TB).
  • Common queries filter on a column that's not the natural insertion order.
  • Query profile shows partition scan ratio is bad.
ALTER TABLE fct_orders CLUSTER BY (order_date, region);

Don't cluster eagerly: auto-clustering costs credits. Verify it pays off via profile.

Streams + Tasks — CDC inside Snowflake

Stream: change tracking on a table. Captures inserts/updates/deletes since last consumed.

Task: scheduled SQL or stored proc. Can be cron'd or chained.

CREATE STREAM orders_stream ON TABLE raw.orders;

CREATE TASK merge_orders
  WAREHOUSE = wh_etl
  SCHEDULE = '1 minute'
WHEN SYSTEM$STREAM_HAS_DATA('orders_stream')
AS
MERGE INTO mart.fct_orders t
USING orders_stream s ON t.order_id = s.order_id
WHEN MATCHED THEN UPDATE SET ...
WHEN NOT MATCHED THEN INSERT ...;

ALTER TASK merge_orders RESUME;

Why use it: sub-minute incremental updates without an external orchestrator.

Snowpipe — auto-ingest from S3
  • S3 ObjectCreated event → SQS → Snowpipe COPYs into table.
  • Latency: ~30s typical from file land to row visible.
  • Charges per file processed (overhead per file). Tune Firehose/upstream to make files reasonable size (5MB+).
  • Alternative: scheduled COPY (cron) — simpler but cron-bounded latency.
Time travel + zero-copy clone — underrated superpowers

Time travel: query a table as of a past timestamp (default 1 day, up to 90d on Enterprise).

SELECT * FROM fct_orders AT(TIMESTAMP => '2025-04-25 10:00:00'::timestamp);
-- Restore: UNDROP TABLE fct_orders; or CREATE TABLE x AS SELECT * FROM y AT(...)

Zero-copy clone: instant table/database/schema clone, no storage cost (only diverging changes cost).

CREATE DATABASE dev_analytics CLONE prod_analytics;

Use cases: dev environments built in seconds, "as of" reporting, accidental delete recovery, A/B testing pipelines.

-- ... change a model, re-run, compare to original ...

Time travel saved us during the AXA dedup investigation: we wanted to see what the staging table looked like 3 days ago before a flag-handling change, to confirm the bug existed before our edit. RBAC the Snowflake way

Roles → users (or other roles). Grants are role-based, not user-based.

  • SYSADMIN — admin of objects (creates DB/schema/warehouse).
  • SECURITYADMIN — manages roles/grants.
  • Custom roles: analytics_read, etl_writer, pii_reader.
  • Functional roles → access roles pattern: functional roles inherit from per-object access roles.
Rule: never grant directly to users. Always to roles. Users get a role.

4. Architecture Senior

Lake vs Warehouse vs Lakehouse — cut through the noise
  • Data Warehouse — structured, schema-on-write, optimized for SQL analytics. Examples: Snowflake, Redshift, BigQuery.
  • Data Lake — raw files in object storage, schema-on-read. Examples: S3 + Glue Catalog + Athena.
  • Lakehouse — warehouse semantics on top of lake storage, via table formats (Iceberg, Delta, Hudi). Examples: Databricks Delta, Snowflake's Iceberg support.

The convergence: warehouses now read external Iceberg tables (Snowflake), lakes have ACID via table formats. The pure lake-vs-warehouse split is fading.

For interviews: state the workload, then pick. "Analytics for BI": warehouse or lakehouse. "Data science with Spark/ML": lakehouse. "Cheap raw archive": lake.

Medallion architecture (Bronze / Silver / Gold)
  • Bronze: raw, append-only, schema preserved. Source of truth, replay layer.
  • Silver: cleaned, conformed, deduplicated. Type-cast, normalized.
  • Gold: business-ready, modeled (facts, dims), aggregated for BI/ML.
When it's overkill: small data, single-source pipelines. Bronze + Gold may be enough. Don't add Silver just to feel architectural.
Real example from healthcare claims work

Cost optimization stories
  • Right-size warehouses: an XS at 60s auto-suspend often beats an L at 5min for sporadic ETL.
  • Partition + cluster: scanning 10% of data is 10x cheaper than 100%.
  • Materialize aggregates: queries that scan 1B rows can hit a 10K-row pre-agg instead.
  • Lifecycle S3: move > 30-day raw data to IA or Glacier.
  • Concurrency scaling only when needed: Snowflake auto-scale clusters cost extra; cap min/max.
  • Kill orphan workloads: BI tools holding sessions, runaway dbt jobs in dev. Monitor.
Data quality frameworks
  • dbt tests: built-in (unique, not_null, relationships, accepted_values), custom singular tests, generic tests via macros.
  • Great Expectations: more expressive (distribution, custom, suite-level), but heavier setup. Use for the "is this data sane?" layer (bronze).
  • Soda / Monte Carlo / Datafold: managed observability, anomaly detection, schema-drift alerts. Pay-for layer.
  • Data contracts: producer-side schema + semantic guarantees, enforced in CI before deploy. Modern shift-left approach.

Freshness SLAs: define them per source. dbt's source freshness alerts if a source is stale.

Real example from healthcare claims work

Data quality has three layers in the claims pipeline:

  1. dbt tests: every model has unique + not_null on PKs, accepted_values on enums (claim_type IN ('institutional','professional')), relationships on FKs (claim.member_id → eligibility.member_id).
  2. Observability models (obs_*): explicit excluded-claim tracking with reasons. obs_medical_claim__excluded answers "where did claim X go?" with the reason (eligibility-filtered, summary-billing, zero-billed, COB, fully-reversed). obs_*__financial_adjustments tracks any amount that changed during processing.
  3. Control totals: data_quality_control_totals compares source row counts and aggregate amounts to final. If source had $1.2M billed and final has $0.9M, the difference must be explainable by exclusion reasons.

The "explainability" requirement is healthcare-specific: claims teams audit changes line-by-line. Every dropped claim needs a logged reason; every modified amount needs a tracked adjustment. Without this, you can't pass a regulatory audit.

30-second pitch
"I owned a multi-tenant healthcare claims ingestion pipeline. Each healthcare partner — insurers, employer groups, third-party administrators — sent us their medical claims, eligibility, and pharmacy data, and we processed it through a Python + dbt + Snowflake pipeline to produce cleaned, deduplicated, finalized claim data for downstream analytics. The flow
1
Source landing: per-tenant claims, eligibility, pharmacy data lands in S3 / Snowflake input schema. One isolated input schema per tenant: INPUT_LAYER_{prospect_id}.
2
SQS event: {"prospect_id": "...", "seed": false} triggers ingest.py in a Python container (Lambda or ECS).
3
Orchestrator: validates env, creates per-tenant Snowflake DB {ENV}_INGEST_{prospect_id}, generates profiles.yml dynamically at /tmp/{prospect_id}/, runs dbt build --threads 8.
4

5

The "why" answers
Why Snowflake over Redshift?
Per-tenant DB isolation is trivial (CREATE DATABASE per tenant + RBAC). Zero-copy clone for debugging gnarly source schemas without touching prod. Per-tenant warehouse scaling without contention. Less ops than Redshift dist/sort key tuning.
Why dbt over stored procs?
Tests at every layer (uniqueness, not-null, custom finalization tests) — the transformation logic is too complex to be a black box. Lineage from raw column through final model. 23 reusable macros for healthcare-specific transforms (NPI cleaning, claim type fix, amount priority lookup).
Why SQS-driven over Airflow / Step Functions?
Tenant ingestion is event-driven, not scheduled. New prospects arrive ad-hoc; existing prospects re-ingest when source data refreshes. Retry + DLQ semantics built in. No scheduler to operate. Trade-off: less visibility into multi-step DAG state — if we needed cross-tenant orchestration, MWAA would've been the call.
Why per-tenant Snowflake database isolation?
HIPAA compliance: cross-tenant queries are physically impossible, not just policy-prevented. Independent ramp-up/teardown. Separate cost attribution. "Delete this tenant" is a one-line DROP DATABASE with Snowflake time travel as the safety net.
Why an open-source healthcare data model?
Standardized terminology + reference data (DRG groupers, ICD/CPT/HCPCS code lookups). Industry-recognized so new hires onboard fast. Trade-off: locked into the framework's modeling choices — we layered custom intermediate models on top rather than fork.
What did you reject?
Single shared warehouse with row-level security (audit story harder; physical separation is simpler compliance). Spark/EMR for transforms (no volume justification). Fivetran/Airbyte (every tenant has a custom source schema; managed connectors don't fit). Real-time ingestion (claims arrive in batches; no business need for sub-hour).
What broke (the debugging stories)

Story 1: Re-adjudicated claims weren't being grouped. A major insurance client sent re-adjudicated claims — same service, same member, same provider, different paid amounts, claim IDs differing by suffix. Symptom: 2 surviving rows in the finalized output instead of 1. Three hypotheses tested. Root cause: a duplicate_claimid_line_number_flag set on all rows (because original line 1 appeared twice) triggered safe line-number renumbering, but a downstream WHERE flag IS NULL filter blocked these claims from finalization entirely. Fix: split the flag into two single-purpose flags — one for renumbering (informational), one for deliberate exclusion. Plus fixed an upstream date-parsing issue where 2025-06-20 00:00:00 was defaulting to 1900-01-01 due to a mapping tool format mismatch.

Lesson: a flag that does two things — protect from one bug, prevent another fix — is a smell. Each flag should have one job.

Story 2: Provider voting was non-deterministic. When a claim had multiple potential rendering NPIs, we voted by state match + provider type match + frequency. Tied votes between an Individual NPI and an Organization NPI were picking different winners across runs. Fix: explicit priority chain (winning > original > voted rendering > voted billing > voted facility) + tiebreaker on NPI string in ORDER BY.

Lesson: voting / ranking logic must be deterministic. ORDER BY needs a tiebreaker even when "ties shouldn't happen".

Story 3: dbt threads, not Snowflake compute, were the bottleneck. Some tenant ingests took 6+ hours when 1-2 hours was expected. Investigation showed dbt's default 4-thread parallelism was leaving the warehouse idle on the long pole. Bumped to --threads 8 for build, --threads 12 for seed. Cut runtime to ~2 hours. Multi-cluster warehouse for high-concurrency tenants.

Lesson: when a pipeline is slow, look at orchestrator parallelism before assuming the warehouse is the bottleneck.

Deep-dive Q&A — be ready for these
Tell me about your SCD2 strategy.
Eligibility is naturally at month grain — one row per member per enrollment month — so we didn't need explicit valid_from/valid_to. The time dimension is the row's grain. For mid-month plan changes, the framework reconciles overlapping period rows. How did you handle PII / PHI?

Service accounts couldn't read raw PHI in prod without explicit override. No flat-file extracts — all reads through Snowflake views. Audit logging native via INFORMATION_SCHEMA.QUERY_HISTORY.

Why claims data is uniquely hard
  • Re-adjudication: a claim paid in March can be re-paid in November with different amounts. Both rows arrive in your pipeline. You have to figure out which is "the truth".
  • Reversal claims: when an original claim is wrong, the payer issues a negative claim that cancels the original, then a new claim with corrected values. Adjustment chains can be 3-4 deep.
  • Source schema variance: every payer / TPA has different column names. allowed_amount in one source is allowed_amt_total in another, allow_amt in a third. Mapping tables + priority lookups handle this.
  • UB vs HCFA: institutional claims use revenue code, no POS. Professional claims use POS, no revenue code. Same dbt model has to handle both gracefully.
  • Provider data is messy: NPIs missing, TINs with dashes, name variations ("Dr. Jane Smith" vs "Smith, Jane MD"). Voting / cleaning logic needed.
  • Eligibility filtering: a claim only counts if the member was enrolled on the date of service. Inner-join with eligibility periods at the start of the pipeline.
  • PHI compliance: cross-tenant data leakage is a regulatory event. Architecture decisions are often driven more by compliance than by performance.
Common claims pipeline patterns (memorize these)

1. Claim line dedup with ROW_NUMBER

SELECT *
FROM (
  SELECT *,
         ROW_NUMBER() OVER (
           PARTITION BY claim_id, claim_line_number, person_id,
                        claim_line_start_date, place_of_service_code,
                        revenue_center_code, hcpcs_code,
                        rendering_npi, paid_amount
           ORDER BY paid_date DESC, paid_amount DESC,
                    allowed_amount DESC, charge_amount DESC
         ) AS rn
  FROM raw.medical_claim
) t
WHERE rn = 1;

HIPAA / PHI — what every healthcare DE should know
  • PHI = Protected Health Information. 18 specific identifiers (name, address, DOB, MRN, SSN, etc.) plus any health info linked to them.
  • De-identification: removing/hashing PHI. Two methods accepted by HIPAA: Safe Harbor (remove all 18 identifiers) and Expert Determination (statistical proof of low re-id risk).
  • Minimum necessary: only access PHI you need. Translates to RBAC: analytics roles see aggregated outputs, engineering roles see structured PHI for pipeline work, no role sees raw flat-file extracts.
  • BAA: Business Associate Agreement. Any vendor handling PHI on the covered entity's behalf must sign one. Snowflake, AWS, dbt Cloud (etc.) all have HIPAA-eligible/BAA-eligible offerings.
  • Audit logs: who queried what, when. Snowflake's QUERY_HISTORY + AWS CloudTrail give you this for free.
  • Data retention: claims data has long retention requirements (often 7-10 years). Build with this in mind — archival tier in S3, time-travel + cold storage in Snowflake.
Healthcare DE interview questions you might get
How would you design a claims warehouse?
Star schema. Fact: fact_claim_line at one row per claim line per service date. Dimensions: dim_member, dim_provider (with rendering / billing / facility roles), dim_date, dim_diagnosis, dim_procedure. Eligibility as a separate fact at member-month grain to support point-in-time joins. SCD2 on member + provider for attribution history. Pre-compute common aggregates (PMPM, utilization rate, paid-per-claim).
A claim has two rendering NPIs. What do you store?
Both, but with one as primary. The "primary" is determined by voting (most frequent across the claim's lines, scored by state match + provider type). The non-primary stays in an array column or related table for audit.
How do you handle re-adjudicated claims?
Group by clinical fingerprint (member, provider, service date, procedure). Within a group, pick the winner by latest paid_date. Net paid_amount across the group must be checked — fully-reversed groups (sum to 0) are excluded from analytics. Keep the full history for audit.
How do you join claims with eligibility?
Inner join on member_id with claim_service_date BETWEEN eligibility_start AND eligibility_end. Claims for non-enrolled members are excluded (or quarantined). For mid-month plan changes, the eligibility table has overlapping rows; pick the row covering the service date.

Why X over Y — quick-fire answers

Practice these out loud in < 60 seconds each. Always include: at least one alternative considered, one reason rejected.

Snowflake vs Redshift
Snowflake separates compute and storage, has multi-warehouse isolation, zero-copy clone for dev environments, and less DBA tuning. Redshift catches up with RA3 + Serverless. For greenfield, Snowflake. For an existing Redshift shop running fine, the migration cost is real — don't move just because.
Airflow vs Step Functions
Airflow for complex Python DAGs with sensors and inter-DAG dependencies. Step Functions for stateless AWS-native workflows, especially Lambda-heavy fan-outs. They often coexist: Step Functions for the orchestration of AWS services, Airflow for data pipelines.
dbt vs Stored Procedures
dbt for version control, tests as code, auto lineage, easier onboarding. Stored procs for deep warehouse-specific tuning or transactional multi-statement logic. dbt has won the analytics layer.
Spark vs pandas vs Snowflake SQL
SQL on the warehouse first — cheapest, fastest if data already lives there. Polars/DuckDB for single-machine work. pandas for interactive analysis < 5GB. Spark only when data doesn't fit on one machine or you're already on a Spark stack.
Lambda vs Glue
Lambda for short (<15min), event-driven, validation/enrichment. Glue for batch ETL with Spark, longer jobs, JSON flattening. Cost crosses over fast: tiny event handler → Lambda, gigabyte-scale transform → Glue.
Kinesis vs Kafka (MSK)
Kinesis: managed, simpler, cheaper at moderate throughput, AWS-native triggers. MSK: open Kafka ecosystem (Connect, KSQL), compaction, longer retention than Kinesis's 7 days. Pick Kinesis unless you need the Kafka ecosystem specifically.
Snowpipe vs scheduled COPY
Snowpipe is event-driven, low-latency (~30s), decouples ingest credits. COPY on cron is simpler but cron-bounded latency. Below sub-minute SLAs → Snowpipe; daily/hourly batches → COPY is fine.
Iceberg vs Delta vs Hudi
Iceberg: vendor-neutral, growing ecosystem, Snowflake/Athena/BigQuery support. Delta: tightest with Databricks, mature. Hudi: best for upsert-heavy CDC scenarios. For greenfield in 2025, Iceberg is the safer bet unless you're Databricks-first.

Debugging walkthroughs

Practice these as 2-min explanations. The structure: symptom → first checks → root cause → fix → prevention.

"A pipeline is slow. Walk me through how you'd diagnose."
  1. Look at the orchestrator first: which task is slow? Is it consistent or new?
  2. Compare to last successful run: same data volume? Same partition? Same code?
  3. If SQL: query profile. Partition scan ratio. Joins reordered? Spills? Bytes scanned?
  4. If Spark: Spark UI. Stage with the long tail. Skew? GC? Shuffle size?
  5. If ingest: source rate-limit, network egress, file count anomaly.
  6. Fix the bottleneck, not the symptom. Bigger warehouse fixes the symptom; the bad clustering key keeps biting you.
"There are duplicate rows in your fact table. What now?"
  1. Define the grain. "One row per X" — what's the unique key?
  2. Find the duplicates: SELECT key, COUNT(*) FROM fct GROUP BY key HAVING COUNT(*) > 1;
  3. Are they bit-identical or different on some columns? Bit-identical = ingest dup. Different = source emitted multiple events.
  4. Trace back: bronze layer dup? Source event dup? Producer retry without idempotency?
  5. Fix at the earliest stage. Idempotency key at producer is gold standard.
  6. Backfill the affected window from raw via your replay path (you have one, right?).
"DAG is stuck in queued. Why?"
  • No worker available. Check pool size, slot count.
  • Sensor blocking. Long-running sensor in a non-deferrable form is hogging a slot.
  • Resource constraint. MWAA size cap reached.
  • Concurrency limit on the DAG or task hit.
  • Trigger rule mismatch (e.g. task waiting on a skipped upstream with all_success).
"Snowflake bill spiked. Why?"
  • WAREHOUSE_METERING_HISTORY: which warehouse, when?
  • QUERY_HISTORY: long-running queries? Wide table scans?
  • Auto-suspend disabled? BI tool holding a session?
  • Materialized view auto-refresh runaway?
  • Snowpipe storm (massive small files)?
  • Concurrency scaling triggered repeatedly during a regression?

4. Retry decorator with exponential backoff 
import functools, time, random

def retry(attempts=3, base=1.0, max_delay=30.0):
    def deco(fn):
        @functools.wraps(fn)
        def wrapper(*args, **kwargs):
            for i in range(attempts):
                try:
                    return fn(*args, **kwargs)
                except Exception:
                    if i == attempts - 1:
                        raise
                    delay = min(base * 2**i, max_delay) + random.random()
                    time.sleep(delay)
        return wrapper
    return deco

import csv, pyarrow as pa, pyarrow.parquet as pq

def transform(row):
    row["amount_cents"] = int(float(row["amount"]) * 100)
    return row

def chunked(iterable, size=10_000):
    chunk = []
    for x in iterable:
        chunk.append(x)
        if len(chunk) == size:
            yield chunk; chunk = []
    if chunk: yield chunk

def etl(in_path, out_path):
    with open(in_path) as f:
        reader = csv.DictReader(f)
        writer = None
        for chunk in chunked(transform(r) for r in reader):
            table = pa.Table.from_pylist(chunk)
            if writer is None:
                writer = pq.ParquetWriter(out_path, table.schema)
            writer.write_table(table)
        if writer: writer.close()

import time
from collections import deque

class RateLimiter:
    def __init__(self, max_calls, per_seconds):
        self.max_calls = max_calls
        self.per = per_seconds
        self.calls = deque()
    def acquire(self):
        now = time.time()
        while self.calls and now - self.calls[0] > self.per:
            self.calls.popleft()
        if len(self.calls) >= self.max_calls:
            time.sleep(self.per - (now - self.calls[0]))
        self.calls.append(time.time())

def safe_get(d, path, default=None):
    """Get nested value: safe_get(d, ['a', 'b', 'c'])."""
    cur = d
    for key in path:
        if not isinstance(cur, dict):
            return default
        cur = cur.get(key, default)
        if cur is default:
            return default
    return cur

16. Idempotent S3 write with atomic rename 
import boto3, uuid

def write_atomic(bucket: str, key: str, body: bytes):
    """Write to a temp key, then copy to the final key. Cleans up on failure."""
    s3 = boto3.client("s3")
    tmp_key = f"{key}.tmp.{uuid.uuid4().hex}"
    try:
        s3.put_object(Bucket=bucket, Key=tmp_key, Body=body)
        s3.copy_object(Bucket=bucket, Key=key,
                       CopySource={"Bucket": bucket, "Key": tmp_key})
    finally:
        s3.delete_object(Bucket=bucket, Key=tmp_key)

Why: a partial write to the final key would leave corrupt data visible to readers. Temp + copy gives you atomic-from-the-reader's-view semantics.

17. Rolling/sliding window without external libraries 
from collections import deque

def rolling_avg(values, window):
    """Yield the rolling average over the last `window` values."""
    buf = deque(maxlen=window)
    total = 0.0
    for v in values:
        if len(buf) == window:
            total -= buf[0]
        buf.append(v)
        total += v
        if len(buf) == window:
            yield total / window

print(list(rolling_avg([1, 2, 3, 4, 5, 6], 3)))  # [2.0, 3.0, 4.0, 5.0]

Rapid-fire trivia (know these cold)

One-liner answers. Drill until you can answer in < 5 seconds.

QuestionAnswer
Difference between WHERE and HAVING?WHERE filters rows before GROUP BY; HAVING filters groups after.
Difference between RANK and DENSE_RANK?RANK leaves gaps after ties (1,1,3); DENSE_RANK doesn't (1,1,2).
Difference between UNION and UNION ALL?UNION dedupes, UNION ALL doesn't. UNION ALL is faster.
Difference between TRUNCATE, DELETE, and DROP?DELETE: row-by-row, can WHERE, can rollback. TRUNCATE: fast wipe, no WHERE. DROP: removes the table itself.
What's a CTE?Common Table Expression — a named subquery defined with WITH.
What's idempotency?Running the same op multiple times produces the same result as once.
What's columnar storage and why is it good for analytics?Stores values column-by-column; queries that touch few columns scan less, compress better.
What's a primary key vs unique key?PK is non-null + unique + only one per table. Unique key is unique but allows nulls and multiple per table.
What's an index?A sorted/structured copy of one or more columns that lets the DB skip full scans for filters/joins.
What's the difference between Parquet and ORC?Both columnar; Parquet is more widely supported (broader ecosystem); ORC has slight edge in Hive-heavy stacks.
What is partitioning?Splitting a table into chunks by a column (usually date) so queries can skip irrelevant chunks.
What is bucketing/clustering?Sub-partitioning by hashing or sorting on a column to co-locate related rows for joins/filters.
SCD2 vs SCD1?SCD2 keeps history with valid_from/valid_to; SCD1 overwrites.
What is CDC?Change Data Capture — pulling INSERT/UPDATE/DELETE events out of an OLTP DB.
What is exactly-once semantics?Each event processed exactly once. Usually achieved via at-least-once delivery + idempotent processing.
What's the difference between batch and streaming?Batch: bounded data, scheduled. Streaming: unbounded, continuous, lower latency.
What's a fact vs a dim table?Fact: events / measurements (orders). Dim: descriptive entities (customer, product). Joined in star schema.
What's grain?"One row per X" — the unique definition of what a row in a fact table represents.
What is a watermark?A timestamp threshold below which all events are considered settled / processed.
What is a DAG?Directed Acyclic Graph — tasks with dependencies, no cycles. Airflow's core abstraction.
What's the difference between a stream and a queue?Queue: each message consumed once (FIFO/SQS). Stream: replayable, multiple consumers (Kafka/Kinesis).
What's denormalization?Combining tables into wider, redundant tables to speed up reads at the cost of write complexity.
What is data skew?One key/partition has way more data than others, causing one task to be slow in distributed systems.
How does Snowflake bill?Compute: per-second of warehouse uptime (credits). Storage: per-TB-month. Cloud services and Snowpipe billed separately.
What's a materialized view?A pre-computed query result stored as a table; auto-refreshed when source changes (in some warehouses).

  3. What we ruled out fast (10s — 1 option that was clearly wrong, to show we knew the landscape)

STAR-shaped, but DE-flavored:

  • Situation — what was broken, who noticed, blast radius.
  • Investigation — the actual diagnostic steps. Show methodology, not just the answer.
  • Root cause — not the surface symptom.
  • Fix — what changed.
  • Prevention — what now ensures it can't happen again.

Trade-off vs Airflow: less DAG visibility (you don't see the staging→intermediate→final breakdown unless you read CloudWatch dbt event logs); simpler retry/DLQ; no scheduler to operate. We considered MWAA when we needed cross-tenant orchestration but defaulted to SQS for tenant-isolated work.

def chunked(it, size):
    chunk = []
    for x in it:
        chunk.append(x)
        if len(chunk) == size:
            yield chunk; chunk = []
    if chunk: yield chunk
